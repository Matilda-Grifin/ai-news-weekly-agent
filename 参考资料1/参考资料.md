1) FinnewsHunter 是怎么爬的（它的“爬虫栈”）
它主要有两套：
A. 站点定制爬虫（各财经站）
在 external/FinnewsHunter/backend/app/tools/*_crawler.py 里（如 sina_crawler.py、eastmoney_crawler.py…）：
按站点写规则：列表页怎么翻页、正文页怎么定位标题/正文/时间
多数用 requests + BeautifulSoup 解析 HTML
对一些站点（如东方财富）标注 needs_js=True，需要 JS 渲染


B. “增强版爬虫”统一正文抽取（值得看）
在 external/FinnewsHunter/backend/app/tools/crawler_enhanced.py：
多引擎：
requests 抓静态 HTML
Playwright 用于 JS 渲染站点
Jina Reader API（可选）用于更稳定地拿正文
内容抽取：readabilipy + 启发式 selector（不同域名一套 selector 配置）
质量评分 + 自动重试：抽出来太短/像噪声就换引擎或重试
统一 Article 模型：把 title/content/html/url/source/publish_time 等统一成结构化对象，方便后续给 LLM
另外它还有一套“搜索引擎补充”思路：
interactive_crawler.py：用 requests 抓 Bing/百度，检测验证码页关键字，被拦就降级
search_engine_crawler.py：解析搜索结果里的链接（但它本质还是爬搜索引擎页面，仍会反爬）


2) 我们现在是怎么爬的（ai_news_skill）
你这个仓库当前的抓取链路是：
站点定制爬虫（优先）：sources.json 里 name 若在 registry 有同名爬虫（如「OpenAI Blog」→ openai.com/news 列表滚动 + 详情页 Playwright），则走 SiteCrawler，且该信源不再走 RSS。
RSS 信源：其余 name 仍用 rss_url → collect_news()（run_daily_digest.py）
GNews API（结构化搜索结果）：fetch_gnews_articles() / fetch_gnews_for_pipeline()
说明：Google HK SERP 自动化检索已从主流程移除（易触发 /sorry）；补充检索请用 GNews 等合规 API。`external/public-apis`（克隆自 public-apis/public-apis）的 **News** 分类中有 Mediastack、NewsAPI、Currents、NewsData、NYTimes、Guardian 等可作信源候选，需各自申请 Key 并自行接进 `collect_stage`。
说明：external/FinnewsHunter 的 requests+BeautifulSoup+needs_js 栈未接入本管线；要复用需按站点再包一层或迁抽取逻辑。
正文兜底抽取：fetch_article_excerpt()（trafilatura + HTML 规则兜底）
attach_content_excerpts_to_items() 在 enrich 前补正文摘录
LLM 富化：enrich_items_with_llm() 生成中文标题/核心内容/应用价值（LLM 失败就 fallback，导致英文漏出）

3) 他有什么我们能复用的（高价值、低风险）
立即可复用（建议做）
“多引擎正文抽取 + 质量评分 + 自动重试”
你的 fetch_article_excerpt 现在是“尽力而为”，但没有“抽取质量差就换引擎/重试”的闭环；FinnewsHunter 这套能显著提升“能读到正文”的比例。
“按域名的 selector 配置”
你现在更多靠 trafilatura；遇到站点结构特殊时会空。它的 FINANCE_SITE_CONFIGS 思路可迁移成你这边的 SITE_CONFIGS，提高命中率。
“统一 Article/Item 模型”
你现在 item 字典字段比较散；引入类似 Article 能让后续 LLM 输入更干净，减少 fallback。
有条件复用（看你是否愿意引入依赖/Key）
Jina Reader API：对“反爬+JS重”的站点更稳定，但需要额外 API Key/费用。
搜索引擎页面爬取：FinnewsHunter 做了验证码检测与降级，但本质仍会反爬（和我们爬 Google 一个问题），不建议作为主力。

4) 一句话对比
FinnewsHunter 强在：正文抽取工程化（多引擎、质量评估、重试、站点配置）。
我们强在：RSS + GNews 这种“相对合规且结构化”的采集，以及后续 LLM 周报生成。
我们最该复用的：把它的“增强版正文抓取闭环”迁到我们这里，专门解决“抓到链接但正文拿不到/太短/质量差”的问题。
